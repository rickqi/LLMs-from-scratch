# 专业领域文本数据获取与训练数据生成 — 完整实施方案

> 基于中文医学文本生成项目（412篇文档 + 607条QA）的完整实践经验提炼。
> 适用于法律、金融、医疗、工程、教育等任何专业领域的 LLM 微调数据准备。
> 2026-06-19

---

## 一、总体架构

专业领域 LLM 微调数据管线分为五个阶段，呈单向流动、逐层加工：

```
┌─────────────────────────────────────────────────────────────────┐
│                     五阶段数据管线                               │
│                                                                 │
│  阶段1          阶段2         阶段3          阶段4       阶段5   │
│  原始数据获取 → 清洗预处理 → 续写训练数据 → 指令QA数据 → 备份  │
│                                                                 │
│  产出:          产出:         产出:          产出:        产出:  │
│  原始文档       标准化文本    train.txt      QA对(JSON)   COS   │
│  (.md/.pdf/.txt) (纯文本)     val.txt        ChatML格式   远程   │
│                               ===SEP===分隔                   │
└─────────────────────────────────────────────────────────────────┘
```

**核心原则**：
- **数据隔离**：原始数据、清洗数据、训练数据分层存储，互不污染
- **可复现**：所有处理步骤脚本化，固定随机种子(seed=42)
- **增量更新**：新增数据可追加处理，已有数据自动跳过
- **双重验证**：续写数据用 loss 评估，指令数据用质量报告检测

## 二、第一阶段：原始数据获取

### 2.1 数据源分级体系

将目标领域的资料按权威性和专业性分为 3-4 级，确保覆盖面的完整性和层次性：

| 层级 | 定义 | 示例（医学） | 示例（法律） |
|:--:|------|------------|------------|

| **L3 专业指南/共识** | 学术团体发布的指南、专家共识、教材 | 中华医学会临床诊疗指南(36分册)、CACA整合指南、CMA专家共识 | 律协业务指引、审判指导意见、法学教材、典型案例汇编 |
| **L4 操作手册/细则** | 实务操作层面的详细规程 | 临床技术操作规范(49分册)、手术操作规程 | 合同范本、诉讼文书格式、尽职调查清单、合规手册 |

### 2.2 获取方式（按可行性排序）

| 方式 | 适用场景 | 工具 | 注意事项 |
|------|---------|------|---------|
| **官网公开下载** | 政府/机构发布的 PDF/HTML | 浏览器手动 | 注意版本号和发布日期 |
| **批量爬虫** | 结构化网站（法规库/裁判文书网） | Python requests + BeautifulSoup | 遵守 robots.txt，设置请求间隔 |
| **API 接口** | 提供开放 API 的数据平台 | REST API SDK | 关注速率限制和认证方式 |
| **第三方数据集** | 学术机构发布的整理后数据 | 直接下载 | 验证数据来源和时效性 |
| **OCR 转换** | 扫描件/图片格式文档 | Tesseract / PaddleOCR | 需人工校对，错误率 1-5% |
| **手动录入** | 无法自动获取的纸质/专有资料 | 人工 | 仅用于高价值核心文档 |

| 用途 | 最低文档量 | 推荐文档量 | 说明 |

| 指令微调（QA） | 300 对 | **500-800 对** | easy:medium:hard ≈ 35:40:25 |
| 验证集 | 训练集 5% | 训练集 5-10% | 必须来自不同文档 |

## 三、第二阶段：数据清洗与预处理

### 3.1 标准清洗管线

```python
def clean_domain_text(text: str) -> str:
    """通用专业文本清洗"""
    # 1. 去除 HTML/XML 标签
    text = re.sub(r'<[^>]+>', '', text)
    # 2. 去除图片/链接
    text = re.sub(r'!\[.*?\]\(.*?\)', '', text)
    text = re.sub(r'https?://\S+', '', text)
    # 3. 去除元数据噪音
    text = re.sub(r'ISBN[^\n]*', '', text)
    text = re.sub(r'页码[^\n]*', '', text)
    text = re.sub(r'第[一二三四五六七八九十\d]+页', '', text)
    # 4. 规范化空白
    text = re.sub(r'\n{4,}', '\n\n', text)      # 压缩多余空行
    text = re.sub(r'[ \t]{2,}', ' ', text)       # 压缩多余空格
    # 5. 去除非正文行
    lines = [l.strip() for l in text.split('\n') if l.strip()]
    return '\n'.join(lines)
```

### 3.2 领域特定的清洗规则

不同领域需要额外处理：

| 领域 | 特定清洗规则 |
|------|------------|
| **法律** | 去掉"第X条"编号但保留正文；识别并保留"款""项"结构 |
| **金融** | 统一数字格式（千分位、小数位数）；去掉表格边框线 |
| **医学** | 保留 ICD/药品通用名；统一中英文术语（CT→计算机断层扫描） |
| **工程** | 保留公式 LaTeX；保留图纸编号引用；统一单位符号 |

### 3.3 质量过滤

| 指标 | 阈值 | 说明 |
|------|:--:|------|
| 最小文本长度 | > 50 字符 | 太短的片段无训练价值 |
| 最大文本长度 | < 120,000 字符 | 单篇过长需分段处理 |
| 有效字符比 | > 60% | 剔除纯符号/乱码文档 |
| 中文占比 | > 50% | 中英文混合文档的处理策略 |

## 四、第三阶段：续写训练数据生成

### 4.1 数据格式

续写训练使用纯文本格式，以 `===SEP===` 分隔每个文档片段：

```
文档1的清洗后文本内容...

===SEP===

文档2的清洗后文本内容...

===SEP===
```

### 4.2 训练/验证集分割

```python
def split_train_val(texts, val_ratio=0.05, seed=42):
    """固定种子的随机分割"""
    rng = random.Random(seed)
    shuffled = list(texts)
    rng.shuffle(shuffled)
    split = int(len(shuffled) * (1 - val_ratio))
    return shuffled[:split], shuffled[split:]

# 保存
save(train_texts, "train.txt")  # ===SEP=== 格式
save(val_texts, "val.txt")
```

### 4.3 关键决策点

| 决策 | 建议 | 理由 |
|------|------|------|
| 按文档分片 vs 按段落分片 | **按文档分片**（短文档）或滑动窗口（长文档） | 保持语义连贯性 |
| 是否保留标题 | **清洗时去掉** | 减少模型对格式的依赖 |
| 多版本去重 | **保留所有版本** | 让模型学习领域知识的时间演化 |
| 训练/验证比例 | **95/5** | 验证集仅用于监控过拟合，非评估 |

## 六、第六阶段：验证与质量保障

### 6.1 续写数据质量指标

| 指标 | 检查方法 | 合格标准 |
|------|---------|:--:|
| 文本完整性 | 随机抽样人工阅读 | 无乱码、无截断 |
| 分隔符正确性 | `grep -c "===SEP===" train.txt` | 分隔符数 = 文档数 - 1 |
| 训练/验证独立性 | 检查 val 中的文档是否在 train 中出现 | 0 重叠 |
| Token 总量 | tokenizer.encode 统计 | 与实际文档量匹配 |

### 6.2 指令数据质量检查清单

每条 QA 逐一检查：

```
□ 问题以问号结尾（？或?）
□ 答案长度 ≥ 50 字符（非敷衍回答）
□ source_text 引用 ≥ 30 字符（有原文依据）
□ keywords 非空（可检索）
□ answer 中无双语混用（除非术语本身为英文）
□ 答案内容不包含 "根据文档" / "原文未提及"（拒绝型回答 → 重新生成）
□ difficulty 和 query_type 分布合理（easy:medium:hard ≈ 35:40:25）
```

### 6.3 自动化质量报告

生成包含以下维度的统计报告：

| 维度 | 指标 |
|------|------|
| 总览 | 总QA数、领域覆盖数、文档来源数、质量问题数 |
| 领域分布 | 各子领域 QA 数量和占比 |
| 难度分布 | easy/medium/hard 的数量和占比 |
| 类型分布 | factual/procedural/diagnostic/comparative 分布 |
| 平均长度 | 问题平均长度、答案平均长度、原文引用平均长度 |
| 样本展示 | 随机 5 条 QA 的完整展示 |

## 七、第七阶段：版本管理与云备份

### 7.1 版本追踪

```
docs/
├── {domain}_qa_cases.json        ← QA 原始数据（最完整的版本）
├── {domain}_qa_report.md         ← 质量分析报告
├── {domain}_instruction_chatml.json   ← ChatML 训练格式
└── {domain}_instruction_alpaca.json   ← Alpaca 训练格式

data_{domain}/
├── train.txt                     ← 续写训练数据
└── val.txt                       ← 续写验证数据
```

每次更新时：
1. 旧版 QA 数据自动扩容（增量追加）
2. 新的 ChatML 导出覆盖旧版（从完整 QA 数据重新生成）
3. 续写数据追加新文档后重新分割 train/val

### 7.2 增量更新流程

```
新增原始文档
  │
  ├── data_prep.py --data_dir {raw} --output_dir {data}
  │   → 追加清洗后的文本到 train/val
  │
  └── qa_generator.py generate --tier medium
      → 只对未处理的新文档生成 QA
      → 追加到现有 QA 数据集
      → export 重新导出完整训练格式
```

### 7.3 云备份方案

```bash
# 环境变量
export COS_SECRET_ID="your_id"
export COS_SECRET_KEY="your_key"

# 全量备份（首次或大更新后）
python backup.py backup

# 增量备份（日常，只上传有变化的文件）
python backup.py backup --incremental

# 选择性备份
python backup.py backup --data-only    # 仅数据
python backup.py backup --model-only   # 仅模型

# 预览（不上传）
python backup.py backup --dry-run
python backup.py list                  # 查看远程文件
```

**备份内容**：所有 `output_*/` LoRA 权重 + `data_*/` 训练数据 + `docs/` QA 数据 + `scripts/` 脚本 + 训练日志

## 八、领域适配实施清单

将本方案迁移到新专业领域时，按以下清单逐项执行：

### 8.1 准备阶段

```
□ 确定领域范围：4 级分类体系（官方→行业→指南→操作）
□ 建立目录结构：{L1_官方}/{L2_行业}/{L3_指南}/{L4_操作}
□ 确定数据格式：优先 .md（纯文本），其次 .txt，避免 .pdf 直接使用
□ 评估数据量：目标 ≥300 篇文档
```

### 8.2 处理阶段

```
□ 编写 clean_{domain}_text() 清洗函数
  □ 领域特定噪音规则（如法律：条文编号；金融：数字格式）
  □ 最小/最大文本长度阈值
□ 运行 data_prep.py 生成续写数据
  □ 验证 train/val 无重叠
  □ 验证 ===SEP=== 分隔符完整
□ 编写 QA 生成 Prompt
  □ 4 种问题类型各 2 个示例
  □ 领域关键词列表
  □ system prompt 定义助手角色
□ 运行 qa_generator.py --tier small 验证流程
  □ 抽样检查 10 条 QA 质量
  □ 调整 Prompt 后运行 --tier medium
□ 导出 ChatML 格式
  □ 拆分独立验证集 (50条 hold-out)
□ COS 云备份
```

### 8.3 验证阶段

```
□ 续写数据：spot-check 随机 20 个片段可读性
□ QA 数据：运行 stats 命令生成质量报告
□ 训练验证：用小模型（0.6B）跑 1 epoch 验证 loss 下降
□ 推理验证：用训练后模型测试 5 个标准 prompt 的生成质量
```
