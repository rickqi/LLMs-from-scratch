### 2.2 加载策略演进

```

      全局统计 11min | 总 I/O: ~72 GB

v2.0: Parquet 缓存 + LazyFrame 复用
      CSV ──[1次扫描]──→ Parquet(54MB, zstd) ──→ 全局统计 <1s
      CSV ──[1次扫描]──→ build_features

v2.1: chunked 模式 (低内存环境)
      feature_lf.sink_parquet() ──→ batch collect → LightGBM 增量
      零全量物化 | 不 OOM
```

### 2.3 Parquet 缓存设计

```python
# 缓存内容: 仅训练期数据 (≤2024-12-31)
# 压缩: zstd level=3
# 列子集: 67列 (排除冗余文本列)
# 结果: 12.16 GB CSV → 54 MB Parquet (225x)
```

**设计原则**：
- 全局统计必须仅使用训练期数据（防止前视偏差）
- 缓存包含全局统计所需的所有列
- `build_features` 需要全量 CSV（val/test 期数据）

## 三、特征工程 71 列设计依据

### 3.1 特征分类及理论基础

| 类别 | 列数 | 示例 | 保险业务依据 |
|------|:---:|------|-------------|
| **身份信息** | 6 | age, age_bucket, insured_gender, main_insured_rela | 不同年龄/性别的疾病谱和费用模式不同 |
| **保单信息** | 12 | duty_code, policy_grp_name, rnew_flag, sale_chnl, pass_months, policy_age_days, monthly_prem | 险种决定保障范围；渠道反映风险选择；续保客户行为不同 |
| **就诊信息** | 8 | hosp_grade, claim_type, fee_item_type, is_public, is_expensive, day_count, est_loss_ratio | 医院等级与费用正相关；费用类型区分门诊/住院/手术 |
| **时序特征** | 5 | month_sin, month_cos, is_yearend, quarter, day_of_week | 理赔有季节性（年末体检高峰、流感季） |
| **目标编码** | 2 | group_code_te, native_place_name_te | 贝叶斯平滑 → 高基数列的稳健编码 |
| **月度滞后** | 17 | monthly_total_lag{1,2,3}, monthly_avg_mom | 时序依赖：历史赔付预测未来 |
| **会员行为** | 10 | member_total_paid, member_avg_claim_size, member_claim_rate | 历史行为最强的未来预测因子（#1 特征） |
| **交叉交互** | 4 | age_bucket_x_duty_code, hosp_grade_x_fee_item_type | 非独立效应：年轻人+门诊 ≠ 老年人+门诊 |
| **金融特征** | 4 | annual_prem_log, pass_prem_log | 保费规模反映保障水平 |
| **大额标记** | 1 | large_case_filled | 大案识别辅助信号 |
| **其他** | 2 | rgt_source, day_count_filled | 数据来源、住院天数 |

### 3.2 关键特征设计决策

**Target Encoding with Bayesian Smoothing**:
```
group_code_te = (group_mean × n_group + global_mean × 10) / (n_group + 10)
```
- 低样本组向全局均值收缩
- 防止少数案例的极端均值主导预测
- 10 是平滑因子（经验值，可通过 Optuna 调优）

**Hash-based Sampling** (确定性采样):
```python
pl.col("case_no").hash(seed=42) % 100 < sample_pct
```
- 同一 `case_no` 永远在同一组
- 训练/预测/回测一致性保证
- 不需要额外存储采样索引

**时序安全分割**:
```python
train: medical_start_date ≤ 2024-12-31
val:   2025-01-01 ≤ date ≤ 2025-06-30
test:  2025-07-01 ≤ date ≤ 2026-06-30
```
- 严格按时间分割，零前视偏差
- 验证集和测试集完全不在训练集中
- 反映真实部署场景：用历史预测未来

## 四、模型选型依据

### 4.1 为什么 LightGBM 而非深度学习

| 维度 | LightGBM | 深度学习 (LSTM/TabNet) |
|------|----------|----------------------|
| 训练速度 | 5-10 min | 2-8 hours |
| 可解释性 | 特征重要性 + SHAP | 黑盒 |
| 表格数据处理 | 原生支持 (直方图算法) | 需要 Embedding + 大量调参 |
| 类别特征 | 原生 Categorical 处理 | 需要手动 Embedding |
| 缺失值 | 自动处理 | 需要填充策略 |
| 小数据表现 | 优秀 | 需要大量数据 |
| 生产部署 | 单文件 .txt | +PyTorch/ONNX 依赖 |

**Kronos 项目的教训**：12 组 BSQ+Transformer 实验全部失败（RankIC ≤ -0.003），最终 LSTM 回归才突破（RankIC +0.205）。对表格数据，简单模型优于复杂模型。

### 4.2 为什么 Tweedie 损失

保险理赔金额的分布特征：
```
     ↑ 概率密度
     │  ┌─ 大量零值 (未理赔/免赔)
     │  │
     │  │    ┌─ 右偏长尾 (大额理赔)
     │  │    │
     │  │    │         ┌────────
     │  │    │         │
     │  │    │         │
     ──┴──┴────┴─────────┴──────→ 金额
       0   1K   10K   100K
```

Tweedie 分布参数 `p ∈ (1, 2)` 同时处理：
- `p → 1` (Poisson)：零值 + 离散计数
- `p → 2` (Gamma)：连续正值 + 右偏
- `p = 1.35` (Optuna 优化)：混合效果最佳

**为什么不用 log-transform + MSE**：
- log(0) = -∞，零值无法处理
- exp(pred) 有偏差（Jensen 不等式）
- 需要 Duan's smearing estimator 修正 → 额外复杂度

### 4.3 分块增量训练失败机理

```
现象: v4.0 chunked (Gini=0.49) < v2.0 sampled (Gini=0.51)
训练数据: 相同 (362万行)
模型结构: 相同 (LightGBM + Tweedie)

根因: keep_training_booster 的局限性

Chunk 1 (500K rows, policies A-E):
  train 50 rounds → Tree 1-50 学习 policies A-E 的模式
  ↓ 冻结

Chunk 2 (500K rows, policies F-J):
  train 50 rounds → Tree 51-100 学习 policies F-J 的模式
  ↓ 但 Tree 1-50 对 policies F-J 的预测无法修改
  ↓ 累积偏差: 树越多偏差越大

最终: 400 棵树 (8 chunks × 50)
     但每 50 棵只对 1/8 的数据有效
     → 等效于 ~50 棵有效树 + 350 棵噪声树
     → 表达能力 ≈ 50 棵树模型 < 35 棵树采样模型

对比单次训练 (25% 采样):
  所有 35 棵树共享同一分布
  → 每棵树对整个数据分布都有贡献
  → 无"冻结树"偏差
```

**LightGBM 增量训练的正确场景**：
- 数据流无限到达（在线学习）
- 数据分布随时间缓慢漂移（概念漂移）
- 每批数据量足够大，分布均匀
- **不适用**：数据按物理存储聚集 → 批量分布不均

## 五、评估指标体系

### 5.1 指标选择依据

| 指标 | 公式 | 为什么选它 | 保险业务含义 |
|------|------|----------|------------|
| **Gini** | 2×AUC(Lorenz) - 1 | 衡量排序能力，不受量纲影响 | "能否区分高风险和低风险案件" |
| **MAE** | mean(\|y-ŷ\|) | 可解释，单位=元 | "平均每个案件预测偏差多少元" |
| **MAPE** | mean(\|y-ŷ\|/y) × 100 | 比例误差，跨保单可比 | "相对误差百分比" |
| **总量误差** | \|Σy-Σŷ\|/Σy × 100 | 保险核心：总赔付预测 | "保费定价和准备金是否充足" |
| **large_auc** | ROC-AUC(>P95) | 大额识别能力 | "能否找出需要调查的大案" |
| **large_recall** | TP/(TP+FN) | 大额查全率 | "大案中有多少被标记" |
| **RankIC** | Spearman(y, ŷ) | 分组排序一致性 | "各组内排序是否一致" |

### 5.2 为什么 Gini > R²

保险理赔的 R² 天然偏低（大量随机性），但 Gini 衡量相对排序：

```
R² 低 ≠ 模型无用

例子: 同一疾病，两个患者费用差 10 倍
      → 随机因素主导 (医生习惯、医院定价)
      → R² 不可能很高
      
但! 如果模型能把 10 倍费用的患者排在前面
    → Gini 可以很高
    → 业务价值: 识别高风险人群进行干预
```

### 5.3 各指标实测值及解读

| 指标 | 基准 | v2.0 采样 | 解读 |
|------|:---:|:---:|------|
| Gini | 0 (随机) | **0.51** | 排序能力中等，需更多数据 |
| MAE | 5,551 (均值基线) | **3,763** | 比基线好 32% |
| R² | 0 | **0.25** | 解释了 25% 方差 |
| 总量误差 | — | **75%** (校正前) | 系统性低估 |
| large_auc | 0.5 (随机) | **0.81** | 大额区分能力强 |
| large_recall | — | **0.10** | 大额查全率低（无专用模型） |

## 六、校准方法论

### 6.1 为什么需要校准

LightGBM Tweedie 回归在保险数据上天然倾向低估：

```
原因 1: Tweedie 损失优化的是条件均值
        → 在长尾分布下，均值被大量小值拉低
        → 对大额案件预测偏低

原因 2: 特征中缺乏金额量级信号
        → 模型学到的模式偏向"平均行为"
        → 离群值被 systematic 地低估

结果: predicted_total / actual_total ≈ 0.25 (低估 4 倍)
```

### 6.2 GroupCalibrator 设计

```python
# 按 policy_grp_name 学习校正因子
factor_g = sum(actual_g) / sum(predicted_g)

# 最小样本保护
if n_group < 30:  # 小保单组用全局因子
    factor_g = global_factor
```

**分组校准优于全局校准的原因**：

```
全局校准: 所有预测 × 3.95
          → 对高估的组也乘 3.95 → 过校正

分组校准: 高估组 × 1.5, 正常组 × 3.0, 低估组 × 6.0
          → 逐组精细调整
```

### 6.3 时间分段校准需求

```
2017 ──────────→ 2025 ──────────→ 未来
  │                │                │
 低估 2x         低估 4x          低估 ?x
  │                │                │
  └── 医疗通胀 ────┘                │
  └── 保单结构变化 ──────────────────┘

解决方案: 按季度训练多个校准器
         cal_2024Q1, cal_2024Q2, ...
         预测时按 medical_start_date 选择对应校准器
```

## 七、L1-B 大额分类器分析

### 7.1 架构设计

```python
LargeClaimClassifier:
  objective: "binary"         # 二分类
  threshold: 24,600 CNY       # P95 分位
  scale_pos_weight: auto      # 自动平衡 (负/正 ≈ 28)
  metric: ["auc", "average_precision"]
```

### 7.2 AUC=0.997 的归因分解

| 因子 | 贡献 | 是否合法 | 机制 |
|------|:---:|:---:|------|
| `member_avg_claim_size` | 主要 (10M gain) | ✅ | 历史大额的会员未来也大额 |
| `age_bucket_x_duty_code` | 主要 (4.7M) | ✅ | 特定年龄+险种→高费用 |
| `large_case_amt` | 代理泄露 | ⚠️ | 与 `y_raw` 正相关 |
| `large_case_filled` | 代理泄露 | ⚠️ | `large_case_amt` 的填充版 |
| `day_count_filled` | 代理泄露 | ⚠️ | `day_count` 的填充版 |
| 二分类放大 | 系统性 | ✅ | 排序中→分离好→AUC高 |

**修正后的真实 AUC**：排除 4 个代理泄露特征后重训 → **AUC=0.9945, AP=0.8729** (仅降 0.0024)。证明模型的排序能力来自合法特征（member_avg_claim_size 等），而非代理泄露。

### 7.3 为什么二分类可以放大排序能力

```
Gini=0.51 的模型在排序上表现：

真实:     [100,  200,  800,  30000, 50000]
预测:     [300,  150,  600,  25000, 45000]
          ─┬──  ─┬──  ─┬──  ──┬───  ──┬───
排序正确:  ✗      ✓     ✓      ✓       ✓     (4/5 正确)

转为二分类 (>24K):
真实:     [0,    0,    0,    1,     1]
预测:     [0,    0,    0,    1,     1]
          ───────── 完全正确 ──────────     AUC=1.0

关键: 只要"真大额排在真小额前面"就行
      金额预测偏差大不影响 AUC
```

## 八、训练模式对比（实测基准）

> 2026-06-21 实测：同一验证集 (173K行)，相同特征 (79列)，相同超参

| 维度 | 全量 (50%) 🏆 | 25% 采样 | 逐保单 |
|------|:---:|:---:|:---:|
| **训练行数** | 1,730,833 | 865,486 | 173,098 (10/批) |
| **Gini** | **0.674** | 0.663 | 0.555 |
| **R²** | **0.905** | 0.863 | 0.580 |
| **MAE** | **2,284** | 2,323 | 3,632 |
| **总量误差** | **9.4%** | 13.0% | 15.9% |
| **large_auc** | **0.975** | 0.970 | 0.924 |
| **树数** | 53 | 45 | 33 |
| **训练时间** | 9s | 9s | 2h |
| **内存峰值** | ~3GB | ~1.5GB | ~500MB |
| **适用场景** | **生产部署** | 快速验证 | ❌ 不推荐 |

### 关键结论

1. **全量 > 采样 > 逐保单**，Gini 最大差 18%。数据量是主导因素。
2. **50% 全量 vs 25% 采样**: Gini 仅差 0.01，但极端案件 (AUC 0.975 vs 0.970) 区分力更强。
3. **逐保单不可用于生产**: 每批 10 保单无法学到完整的 `policy_grp_name` 分布 (SHAP top-5 特征)。
4. **全量训练边际收益递减**: 50%→100% 预期 Gini 仅 +0.01~0.02，但内存翻倍。**50% 是性价比最优**。
5. **Gap 监控生效**: 全量训练 53 棵树，无过拟合 (gap未触发)。
