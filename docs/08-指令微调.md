## 二、核心知识点

### 2.1 指令微调 vs 分类微调

第 6 章我们做了分类微调（垃圾短信检测），第 7 章做指令微调。两者有什么区别？

| 维度 | 分类微调 | 指令微调 |
|------|---------|---------|
| **输出** | 单个类别标签（如 spam/not spam） | 完整文本回复 |
| **训练目标** | 交叉熵（分类，预测标签） | 交叉熵（生成，逐 token 预测） |
| **数据格式** | (文本, 标签) | (指令, 输入, 期望输出) |
| **模型修改** | 替换输出头（Linear 层） | **不修改架构** |
| **用途** | 特定任务 | 通用对话 |

**关键问题：为什么指令微调不需要修改模型架构？**

因为指令微调本质上仍然是 next-token prediction。模型的任务还是"给定前文，预测下一个 token"，只是训练数据从无标注的纯文本变成了"指令-回复"对。模型通过学习在指令上下文中生成合适的回复来掌握遵循指令的能力。

第 6 章和第 7 章是**独立**的——都从预训练模型出发，分别做分类微调和指令微调，互不依赖。

### 2.2 指令数据格式

本书使用 **Alpaca 风格** 的指令数据格式。原始数据以 JSON 存储：

```json
{
  "instruction": "Identify the correct spelling of the given word.",
  "input": "Occassion",
  "output": "The correct spelling is 'Occasion.'"
}
```

三条字段：
- `instruction`：任务描述（必填）
- `input`：额外上下文（可选，可为空字符串）
- `output`：期望的回复（训练目标）

**格式化函数**将 JSON 条目转为模型输入的 prompt 模板：

```python
def format_input(entry):
    instruction_text = (
        f"Below is an instruction that describes a task. "
        f"Write a response that appropriately completes the request."
        f"\n\n### Instruction:\n{entry['instruction']}"
    )
    input_text = f"\n\n### Input:\n{entry['input']}" if entry["input"] else ""
    return instruction_text + input_text
```

完整的训练文本 = `format_input(entry)` + `\n\n### Response:\n` + `entry['output']`：

```
Below is an instruction that describes a task. Write a response
that appropriately completes the request.

### Instruction:
Identify the correct spelling of the given word.

### Input:
Occassion

### Response:
The correct spelling is 'Occasion.'
```

模型只学习生成 `### Response:` 之后的内容。

### 2.3 数据集准备

#### 数据加载与拆分

从 JSON 文件加载指令数据，按 85/10/5 的比例拆分：

```python
file_path = "instruction-data.json"
url = ("https://raw.githubusercontent.com/rasbt/LLMs-from-scratch/main/"
       "ch07/01_main-chapter-code/instruction-data.json")
data = download_and_load_file(file_path, url)

train_portion = int(len(data) * 0.85)   # 935 条
test_portion = int(len(data) * 0.1)     # 110 条
train_data = data[:train_portion]
test_data = data[train_portion:train_portion + test_portion]
val_data = data[train_portion + test_portion:]  # 55 条
```

#### InstructionDataset 类

将每条数据预 tokenize 为 token ID 序列：

```python
class InstructionDataset(Dataset):
    def __init__(self, data, tokenizer):
        self.data = data
        self.encoded_texts = []
        for entry in data:
            instruction_plus_input = format_input(entry)
            response_text = f"\n\n### Response:\n{entry['output']}"
            full_text = instruction_plus_input + response_text
            self.encoded_texts.append(tokenizer.encode(full_text))

    def __getitem__(self, index):
        return self.encoded_texts[index]

    def __len__(self):
        return len(self.data)
```

注意：每条数据的长度不同，所以不能直接组成 batch——需要 padding。

#### Custom Collate Function（核心）

自定义的 collate 函数处理三件事：**padding、target mask、长度截断**。

```python
def custom_collate_fn(batch, pad_token_id=50256, ignore_index=-100,
                      allowed_max_length=None, device="cpu"):
    batch_max_length = max(len(item)+1 for item in batch)
    inputs_lst, targets_lst = [], []

    for item in batch:
        new_item = item.copy()
        new_item += [pad_token_id]  # 添加 eos token
        padded = new_item + [pad_token_id] * (batch_max_length - len(new_item))
        inputs = torch.tensor(padded[:-1])
        targets = torch.tensor(padded[1:])

        # 关键：只保留第一个 padding token 作为 eos，其余标记为 ignore
        mask = targets == pad_token_id
        indices = torch.nonzero(mask).squeeze()
        if indices.numel() > 1:
            targets[indices[1:]] = ignore_index

        if allowed_max_length is not None:
            inputs = inputs[:allowed_max_length]
            targets = targets[:allowed_max_length]

        inputs_lst.append(inputs)
        targets_lst.append(targets)

    inputs_tensor = torch.stack(inputs_lst).to(device)
    targets_tensor = torch.stack(targets_lst).to(device)
    return inputs_tensor, targets_tensor
```

**`-100` 的作用**：PyTorch 的 `cross_entropy` 函数默认将标签为 `-100` 的位置**忽略**，不计算 loss。这样：
- Padding token 不影响训练
- 只有 response 部分的 token 参与 loss 计算

#### 创建 DataLoader

```python
tokenizer = tiktoken.get_encoding("gpt2")
device = torch.device("cuda" if torch.cuda.is_available() else "cpu")

customized_collate_fn = partial(
    custom_collate_fn, device=device, allowed_max_length=1024
)

train_dataset = InstructionDataset(train_data, tokenizer)
train_loader = DataLoader(
    train_dataset, batch_size=8,
    collate_fn=customized_collate_fn,
    shuffle=True, drop_last=True
)

val_dataset = InstructionDataset(val_data, tokenizer)
val_loader = DataLoader(
    val_dataset, batch_size=8,
    collate_fn=customized_collate_fn,
    shuffle=False, drop_last=False
)
```

### 2.5 模型评估

分类微调可以直接算准确率，但指令微调生成的是自由文本，如何评估？

#### 方法一：LLM-as-a-Judge（自动化）

用另一个 LLM（如 Llama 3）给模型回复打分：

```python
def generate_model_scores(json_data, json_key, model="llama3"):
    scores = []
    for entry in tqdm(json_data, desc="Scoring entries"):
        prompt = (
            f"Given the input `{format_input(entry)}` "
            f"and correct output `{entry['output']}`, "
            f"score the model response `{entry[json_key]}`"
            f" on a scale from 0 to 100, where 100 is the best score. "
            f"Respond with the integer number only."
        )
        score = query_model(prompt, model)
        scores.append(int(score))
    return scores
```

运行方式：

```bash
# 先启动 Ollama
ollama serve

# 运行评估
python ollama_evaluate.py --file_path instruction-data-with-response.json
```

典型输出：
```
Average score: 51.75
```

这种方式类似 **AlpacaEval** 评分——用更强的模型评估弱模型的输出。

#### 方法二：人工评估（Manual Review）

直接阅读生成的回复，判断质量：
- 回复是否正确？
- 是否有语法错误？
- 是否完整回答了问题？

实际项目中两种方法通常结合使用。

### 2.6 对齐（Alignment）：DPO 和 RLHF

指令微调让模型学会遵循指令，但模型输出的"风格"和"安全性"还需要进一步优化——这就是**对齐**。

对齐的目标：让模型输出符合人类偏好（**有用、无害、诚实**）。

| 方法 | 原理 | 特点 |
|------|------|------|
| **RLHF** | 训练奖励模型 + PPO 强化学习 | 经典方法，训练复杂（三个模型） |
| **DPO** | 直接在偏好数据对上优化 | 无需奖励模型，实现简单 |

#### RLHF（Reinforcement Learning from Human Feedback）

流程：
1. 用人类标注的偏好数据训练一个**奖励模型**（Reward Model）
2. 用 PPO（Proximal Policy Optimization）强化学习优化 LLM
3. LLM 生成的回复越受人类偏好，奖励越高

缺点：训练过程复杂，需要同时维护三个模型（LLM、Reward Model、Reference Model）。

#### DPO（Direct Preference Optimization）

核心思想：**好回复的概率应该比坏回复高**。

训练数据格式：
```json
{
  "instruction": "Convert active to passive: 'The chef cooks every day.'",
  "chosen": "The meal is cooked by the chef every day.",
  "rejected": "Cooking is done by the chef."
}
```

DPO 直接在偏好数据对上优化 LLM，无需单独的奖励模型。损失函数的直觉：

```
L_DPO = -log(σ(β * (log π(y_good|x) / π_ref(y_good|x)
                      - log π(y_bad|x)  / π_ref(y_bad|x))))
```

- `π`：当前模型
- `π_ref`：参考模型（微调前的模型）
- `σ`：sigmoid 函数
- `β`：控制偏好的强度

本书在 `ch07/04_preference-tuning-with-dpo/dpo-from-scratch.ipynb` 中从零实现了 DPO。

## 四、常见问题

**Q: 指令微调为什么不修改模型架构？**
A: 指令微调本质仍是 next-token prediction，训练数据从无标注文本变为"指令-回复"对。模型学习的是在给定指令上下文时，生成什么样的回复。因此不需要修改输出头或任何架构——和预训练使用完全相同的模型。

**Q: 为什么只对 response 部分计算 loss？**
A: 模型需要学会的是"如何回答"，而不是"如何重复指令"。如果对 instruction 部分也计算 loss，模型会浪费大量容量在学习复制输入上，降低回复质量。PyTorch 用 `-100` 标签让 `cross_entropy` 忽略特定位置。

**Q: 指令数据集从哪里来？**
A: 三种主要来源：
1. **人工编写**：质量最高但成本高（如 Alpaca 最初用 GPT-3.5 生成）
2. **LLM 生成**：用更强的模型（如 GPT-4、Llama 3）生成指令-回复对
3. **开源数据集**：如 Alpaca、Dolly、Open-Orca 等

本书使用约 1100 条预生成的 Alpaca 风格数据。`ch07/05_dataset-generation/` 提供了用 Llama 3 生成数据集的代码。

**Q: 怎么评估生成质量？**
A: 自由文本无法用简单指标衡量。本书介绍两种方法：
1. **LLM-as-a-Judge**：用另一个 LLM（如 Llama 3）给回复打 0-100 分
2. **人工评估**：直接阅读判断质量

工业界还有 BLEU、ROUGE 等指标，但和人类判断相关性有限。

**Q: DPO 和 RLHF 哪个好？**
A: 没有绝对优劣：
- **DPO** 更简单实用，无需训练奖励模型，近年越来越流行
- **RLHF** 理论上更灵活（可以迭代收集偏好数据），但实现复杂

对大多数应用场景，DPO 是推荐起点。

**Q: 指令微调需要多少数据？**
A: 取决于任务复杂度和基础模型能力。经验上：
- 几百到几千条高质量数据就能让模型学会基本的指令跟随
- 本书使用约 1100 条（935 训练 + 55 验证 + 110 测试）
- 工业界通常使用 10K-100K 条数据

数据质量比数量更重要。
