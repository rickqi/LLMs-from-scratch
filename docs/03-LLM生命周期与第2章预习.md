# Part A：LLM 开发生命周期

## 一、使用 LLM 的三种方式

在动手构建之前，先理解我们日常接触 LLM 的三种场景：

| 方式 | 例子 | 特点 |
|------|------|------|
| **公共 API 服务** | ChatGPT、Perplexity、Gemini | 最方便，但不能定制 |
| **本地运行开源模型** | Llama 3、Mistral（通过 Ollama/litellm） | 数据不出本地，可微调 |
| **从零构建自己的模型** | **← 本书做的事** | 理解底层原理，教育价值最大 |

> 本书选择方式三：从零手写 GPT，不是为了造一个比 GPT-4 更强的模型，而是**理解每个组件如何工作**。

## 二、LLM 开发的三个阶段

### 阶段一：构建与预训练（Stage 1 — Building & Pretraining）

```
原始文本 → 分词 → Token ID → 嵌入 → 注意力 → GPT 架构 → 预训练
```

- **数据准备**：收集大规模文本语料（书籍、网页、代码等）
- **分词（Tokenization）**：将文本拆分为 token，映射为整数 ID
- **模型架构**：搭建 Transformer/GPT 网络
- **预训练**：在大规模无标签文本上训练，学习通用语言表示
- **目标**：给定前文 token，预测下一个 token（自回归）

> 本书第 2-5 章覆盖这一阶段。（其中第 4 章用绝对位置嵌入实现 GPT-2，ch05/07_gpt_to_llama 补充目录展示了 LLaMA 使用的 RoPE 旋转位置编码。）

### 阶段二：监督微调（Stage 2 — Supervised Finetuning）

```
预训练模型 + 任务特定数据 → 微调 → 任务专家模型
```

两种微调范式：

| 类型 | 输出 | 数据格式 | 本书章节 |
|------|------|---------|---------|
| **分类微调** | 类别标签（如"垃圾/非垃圾"） | `(文本, 标签)` | 第 6 章 |
| **指令微调** | 完整文本回复 | `(指令, 输入, 期望回复)` | 第 7 章 |

> 预训练模型已具备语言理解能力，微调只是让它适应特定任务。

### 阶段三：对齐（Stage 3 — Alignment）

```
微调模型 + 人类偏好数据 → RLHF/DPO → 符合人类偏好的模型
```

- **目标**：让模型输出更有用、更安全、更符合人类期望
- **方法**：
  - **RLHF**（Reinforcement Learning from Human Feedback）— 训练奖励模型 + PPO 强化学习
  - **DPO**（Direct Preference Optimization）— 直接在偏好数据对上优化，无需奖励模型

> 本书附录 E（LoRA）和第 7 章的 bonus 部分涉及对齐技术。

## 五、核心知识点

### 5.1 词嵌入（Word Embeddings）— Section 2.1

**概念**（无代码）：将文本转化为高维稠密向量。

- 相似的词 → 相似的向量（"cat" 和 "dog" 的向量接近）
- LLM 通过嵌入层将离散的 token 映射为连续的向量空间
- 这一步是后续所有计算的基础

---

### 5.2 文本分词（Tokenization）— Section 2.2

**将原始文本拆分为单词和标点符号。**

```python
import re

# 用正则表达式拆分文本
result = re.split(r'([,.:;?_!"()\']|--|\s)', raw_text)
# 过滤空字符串
tokens = [t for t in result if t.strip()]
```

**示例**：
```
输入: "Hello, world. This is a test."
输出: ['Hello', ',', ' world', '.', ' This', ' is', ' a', ' test', '.']
```

> 本章使用 Edith Wharton 的短篇小说 "The Verdict"（164 行）作为练习语料。

### 5.3 词表与 Token ID — Section 2.3

**构建词表（Vocabulary）：token 字符串 ↔ 整数 ID 的双向映射。**

```python
# 构建词表
all_tokens = sorted(set(tokens))
vocab = {token: i for i, token in enumerate(all_tokens)}
# vocab = {'!': 0, '"': 1, ',': 2, ... 'the': 1132, ...}

# 编码：文本 → ID
def encode(text):
    return [vocab[t] for t in tokenize(text)]

# 解码：ID → 文本
def decode(ids):
    return " ".join([inverse_vocab[i] for i in ids])
```

### 5.4 特殊 Token — Section 2.4

为词表添加特殊标记：

| Token | 用途 |
|-------|------|
| `<\|unk\|>` | 未知词（不在词表中的 token） |
| `<\|endoftext\|>` | 文档分隔符（标记不同文档的边界） |

```python
# 添加到词表末尾
vocab["<|unk|>"] = len(vocab)
vocab["<|endoftext|>"] = len(vocab)
```

### 5.5 BPE 分词器（Byte Pair Encoding）— Section 2.5

**用 `tiktoken` 库替代手写分词器。**

BPE 是 GPT-2/3/4 使用的分词算法：

```
1. 将文本拆分为字节级基础字符
2. 统计相邻 token 对的出现频率
3. 合并最高频的 token 对 → 形成新 token
4. 重复 2-3 直到达到目标词汇量
```

```python
import tiktoken

tokenizer = tiktoken.get_encoding("gpt2")

# 编码
ids = tokenizer.encode("Hello, world")
# [15496, 11, 995]

# 解码
text = tokenizer.decode([15496, 11, 995])
# "Hello, world"
```

**BPE 的优势**：能处理未见过的词（拆分为子词单元），不像简单词表那样只能标记为 `<|unk|>`。

> GPT-2 的词汇量 = 50,257 个 token。

### 5.6 滑动窗口数据采样 — Section 2.6

**⭐ 本章最核心的部分：创建训练用的 input-target 对。**

LLM 的任务是"预测下一个 token"，因此 target 就是 input 向右偏移一位：

```
原始 token 序列: [t1, t2, t3, t4, t5, t6, t7, t8, ...]

max_length=4, stride=2:
  ┌─ input:  [t1, t2, t3, t4]    target: [t2, t3, t4, t5]
  ┌─ input:  [t3, t4, t5, t6]    target: [t4, t5, t6, t7]
  └─ input:  [t5, t6, t7, t8]    target: [t6, t7, t8, t9]
```

```python
import torch
from torch.utils.data import Dataset, DataLoader

class GPTDatasetV1(Dataset):
    def __init__(self, txt, tokenizer, max_length, stride):
        self.input_ids = []
        self.target_ids = []
        token_ids = tokenizer.encode(txt)

        for i in range(0, len(token_ids) - max_length, stride):
            input_chunk = token_ids[i:i + max_length]
            target_chunk = token_ids[i + 1:i + max_length + 1]
            self.input_ids.append(torch.tensor(input_chunk))
            self.target_ids.append(torch.tensor(target_chunk))

    def __len__(self):
        return len(self.input_ids)

    def __getitem__(self, idx):
        return self.input_ids[idx], self.target_ids[idx]

def create_dataloader(txt, batch_size=4, max_length=256,
                      stride=128, shuffle=True):
    tokenizer = tiktoken.get_encoding("gpt2")
    dataset = GPTDatasetV1(txt, tokenizer, max_length, stride)
    return DataLoader(dataset, batch_size=batch_size, shuffle=shuffle)
```

**关键参数**：

| 参数 | 含义 | 典型值 |
|------|------|--------|
| `max_length` | 每个输入序列的长度 | 256 |
| `stride` | 滑动窗口步长（越小重叠越多） | 128 |
| `batch_size` | 每个 batch 的样本数 | 4 |

### 5.7 Token 嵌入（Token Embeddings）— Section 2.7

**将整数 Token ID 映射为稠密向量。**

```python
import torch.nn as nn

# 词汇表大小=50257, 嵌入维度=768
tok_emb = nn.Embedding(50257, 768)

# token_ids: [15496, 11, 995] → 三个 768 维向量
token_embeddings = tok_emb(token_ids)
# shape: (3, 768)
```

> 嵌入层本质上是一个查找表（lookup table），第 i 行就是 token ID=i 的向量表示。

---

### 5.8 位置编码（Positional Encoding）— Section 2.8

**将位置信息注入嵌入，让模型知道每个 token 的位置。**

```python
# 位置嵌入（最大长度=1024, 维度=768）
pos_emb = nn.Embedding(1024, 768)

# 位置 [0, 1, 2] → 三个 768 维位置向量
positions = torch.arange(3)
positional_embeddings = pos_emb(positions)

# 最终输入 = token 嵌入 + 位置嵌入
input_embeddings = token_embeddings + positional_embeddings
# shape: (3, 768)
```

```
Token Embedding:    [0.2, -0.5, 0.8, ...]  ← 编码"这个词是什么"
Positional Embedding: [0.1, 0.3, -0.2, ...]  ← 编码"这个词在第几个位置"
                        ↓ 相加
Input Embedding:    [0.3, -0.2, 0.6, ...]  ← 同时包含语义和位置信息
```

> GPT 使用**可学习的位置嵌入**（不是固定的正弦编码），位置向量在训练中更新。

## 十、常见问题

**Q: `the-verdict.txt` 在哪里？**
A: 在 `ch02/01_main-chapter-code/the-verdict.txt`。如果 notebook 中下载失败，文件已在仓库中提供。

**Q: tiktoken 支持哪些编码？**
A: GPT-2 使用 `"gpt2"`，GPT-4 使用 `"cl100k_base"`。本书全程使用 `"gpt2"` 编码。

**Q: stride 设置多大合适？**
A: 通常 `stride = max_length / 2` 是好的折中。stride 太小会产生大量冗余数据，太大则浪费训练机会。本书默认 `max_length=256, stride=128`。

**Q: 为什么要用位置嵌入？Transformer 不是本来就知道位置吗？**
A: 不是。自注意力机制是"位置无关"的——它只看 token 之间的关系，不知道谁在前谁在后。如果不加位置信息，"猫追狗"和"狗追猫"对模型来说是一样的。

**Q: BPE 和 WordPiece 有什么区别？**
A: BPE 合并最高频的字符对，WordPiece 选择使似然最大的字符对。GPT 系列用 BPE，BERT 用 WordPiece。实际效果相似。

<summary><b>Q7: `nn.Embedding(50257, 768)` 做了什么？和 `nn.Linear` 有什么关系？</b></summary>

**A:** 创建一个 50257×768 的查找表。输入 token ID（整数），输出对应的 768 维向量。

本质上等效于：对 one-hot 向量做矩阵乘法 `one_hot(id) @ weight_matrix`。但 Embedding 层直接索引，不需要实际构造 one-hot 向量，效率更高。

</details>

<details>
<summary><b>Q8: 第 2 章完成后，数据从原始文本到模型输入经历了哪些步骤？</b></summary>

**A:**
```
原始文本 "The cat sat"
  → 分词 (tiktoken): [464, 3797, 3290]
  → 滑动窗口采样: input=[464, 3797], target=[3797, 3290]
  → Token Embedding: 每个 ID → 768 维向量
  → + Positional Embedding: 每个位置 → 768 维向量
  → input_embeddings: shape (2, 768) — 可直接送入 Transformer
```

</details>

<details>
<summary><b>Q9: 为什么本书使用 TensorFlow 而不是 PyTorch 来加载 GPT-2 权重？</b></summary>

**A:** OpenAI 最初发布 GPT-2 预训练权重时使用的是 TensorFlow 格式。本书在第 5 章需要读取这些权重文件，因此需要 TensorFlow。读取后会将权重转换为 PyTorch 张量，后续所有操作都在 PyTorch 中完成。

</details>
