## 一、微调的两条路线

在 LLM 的生命周期中，微调分为两种：

| 类型 | 目标 | 章节 | 输出 |
|------|------|------|------|
| **分类微调** | 预测类别标签 | 第 6 章（本章） | ham / spam（2 个类别） |
| **指令微调** | 跟随指令生成文本 | 第 7 章 | 自由文本回答 |

```
┌──────────────────────────────────────────────────────────┐
│                    预训练模型（基础）                        │
│                只会"续写"文本，不能回答问题                   │
└─────────────┬────────────────────────┬───────────────────┘
              │                        │
    ┌─────────▼──────────┐   ┌────────▼──────────┐
    │   分类微调（Ch6）    │   │  指令微调（Ch7）   │
    │  输出：类别标签      │   │  输出：文本回答     │
    │  例：ham / spam     │   │  例：总结/翻译/回答 │
    └────────────────────┘   └───────────────────┘
```

**为什么先学分类微调？**
- 更简单，是微调概念的入门
- 小型 LLM 的经典应用场景（客服情感分析、垃圾邮件过滤、新闻分类等）
- 不需要通用聊天机器人——很多商业场景只需要一个预测值

## 二、数据集准备

### 2.1 数据集：SMS Spam Collection

来自 UCI 机器学习仓库的公开数据集，包含约 **7,000 条** SMS 短信，每条标注为 `ham`（正常）或 `spam`（垃圾）。

```python
# 数据集是 TSV（制表符分隔）格式，没有表头
# 结构：Label \t Text
# ham     Go until jurong point, crazy..
# spam    Free entry in 2 a wkly comp...
```

**加载数据**：

```python
import pandas as pd

df = pd.read_csv(
    filepath_or_buffer="sms_spam_collection/SMSSpamCollection",
    sep="\t",          # 制表符分隔
    header=None,       # 没有表头
    names=["Label", "Text"]
)
```

### 2.2 类别平衡处理

数据集存在类别不平衡（ham 远多于 spam），需要对 ham 类进行**下采样**使两类数量相等：

```python
# 下采样使 ham 和 spam 数量一致
min_count = df["Label"].value_counts().min()
df_balanced = pd.concat([
    df[df["Label"] == "ham"].sample(min_count, random_state=42),
    df[df["Label"] == "spam"].sample(min_count, random_state=42)
])
```

### 2.3 标签编码

将字符串标签映射为整数：

```python
df_balanced["Label"] = df_balanced["Label"].map({"ham": 0, "spam": 1})
```

| 原始标签 | 整数编码 |
|----------|---------|
| ham（正常） | 0 |
| spam（垃圾） | 1 |

### 2.4 数据集划分

**先打乱，再按比例划分**：

```
┌──────────────────────────────────────────────────┐
│  打乱后的数据集（全部样本随机排列）                    │
├───────────────────────┬──────┬────────────────────┤
│      训练集 70%        │ 验证  │      测试集        │
│                       │ 10%  │      20%           │
└───────────────────────┴──────┴────────────────────┘
```

```python
# 打乱数据
df_shuffled = df_balanced.sample(frac=1, random_state=42)

# 计算划分索引
train_frac = 0.7
val_frac = 0.1

train_end = int(len(df_shuffled) * train_frac)
val_end = train_end + int(len(df_shuffled) * val_frac)

train_df = df_shuffled[:train_end]
val_df = df_shuffled[train_end:val_end]
test_df = df_shuffled[val_end:]
```

> **为什么测试集比验证集大？** 测试集给出最终评估结果，更重要一些，所以分配更多数据。验证集主要用于训练过程中的超参数调优。

### 3.1 Tokenizer

沿用前面章节的 BPE 分词器（`tiktoken`），与预训练时一致：

```python
import tiktoken

tokenizer = tiktoken.get_encoding("gpt2")
```

### 3.2 SpamDataset 类

每条短信长度不同，需要统一长度：

```
短消息：  "Hi"           → [1724, 50256, 50256, ..., 50256]  → 填充到 120
中等消息："Call me now"  → [1724, 502, ..., 50256, ..., 50256] → 填充到 120  
长消息：  "Win free..."  → [1724, 502, ..., 1724, 502, 1724]  → 截断到 120
```

**关键设计决策**：
- `max_length` 由**训练集最长样本**决定（结果为 **120 tokens**）
- 验证集和测试集使用训练集的 `max_length`（截断或填充）
- 填充使用 `<|endoftext|>` token（ID = 50256）

```python
class SpamDataset(Dataset):
    def __init__(self, csv_file, tokenizer, max_length=None, pad_token_id=50256):
        self.data = pd.read_csv(csv_file)
        self.encoded_texts = [
            tokenizer.encode(text) for text in self.data["Text"]
        ]

        if max_length is None:
            self.max_length = max(len(t) for t in self.encoded_texts)
        else:
            self.max_length = max_length
            # 截断超长序列
            self.encoded_texts = [t[:self.max_length] for t in self.encoded_texts]

        # 填充短序列
        self.encoded_texts = [
            t + [pad_token_id] * (self.max_length - len(t))
            for t in self.encoded_texts
        ]

    def __getitem__(self, idx):
        encoded = torch.tensor(self.encoded_texts[idx])
        label = self.data.iloc[idx]["Label"]
        return encoded, label
```

> **为什么不看验证/测试集来确定 max_length？** 这是最佳实践——只能使用训练集的信息，否则相当于"偷看"了测试数据。

## 五、修改模型架构

### 5.1 替换输出头

这是本章最关键的一步。原始 GPT 模型的输出层：

```
原始架构：
  输入 → Token Embedding → 12 个 Transformer Blocks → LayerNorm → 输出层
                                                                    ↓
                                                              768 → 50,257
                                                            （预测下一个 token）
```

**替换后的架构**：

```
修改后：
  输入 → Token Embedding → 12 个 Transformer Blocks → LayerNorm → 新输出层
                                                                    ↓
                                                                768 → 2
                                                           （ham / spam 分类）
```

```python
# 冻结所有参数（可选，后面会解冻部分层）
for param in model.parameters():
    param.requires_grad = False

# 替换输出层
num_classes = 2
model.out_head = torch.nn.Linear(
    in_features=768,      # 嵌入维度（GPT-2 small）
    out_features=num_classes  # 类别数
)
```

> **为什么输出节点数 = 类别数？** 这是通用设计——如果做新闻分类（商业/体育/政治/财经），就设 `out_features=4`。二分类技术上只需 1 个输出，但用 2 个更通用。

### 5.2 解冻最后几层

最佳实践是解冻**最后一个 Transformer Block + 最终 LayerNorm + 输出头**：

```python
# 解冻最后一个 Transformer Block
for param in model.trf_blocks[-1].parameters():
    param.requires_grad = True

# 解冻最终的 LayerNorm
for param in model.final_norm.parameters():
    param.requires_grad = True

# 输出头已经是可训练的（新创建的层默认 requires_grad=True）
```

### 5.3 冻结策略对比

Sebastian 展示了不同冻结策略的实验结果（基于类似分类任务）：

| 训练的层 | 准确率 | 训练时间 |
|----------|--------|----------|
| 仅输出头 | ~76% | ~2.5 分钟 |
| 输出头 + 最后 1 个 Transformer Block | ~87.5% | ~2.5 分钟 |
| 输出头 + 最后 2 个 Transformer Blocks | ~90% | ~3-5 分钟 |
| 所有层 | ~90% | ~7 分钟 |

```
准确率
  │
  │                    ┌──────┐ ┌──────┐
  │               ┌────┤      ├─┤      │  ← 收益递减
  │          ┌────┤    │      │ │      │
  │     ┌────┤    │    │      │ │      │
  │─────┤    │    │    │      │ │      │
  └─────┴────┴────┴────┴──────┴─┴──────┴─── 训练的层数
  只训练   +1层   +2层  +3层    全部
  输出头
```

**结论**：
- **数据集越简单**，训练更多层的收益越小
- 训练输出头 + 最后一个 Transformer Block 是最佳性价比
- 训练全部层时间翻 3 倍，但准确率提升有限

## 六、分类损失与准确率

### 6.1 为什么用最后一个 Token？

由于**因果注意力掩码**（Causal Attention Mask），每个 token 只能看到自己和之前的 token：

```
输入:  "Do"  "you"  "have"  "time"
        │     │      │       │
  Do →  ●     -      -       -      只看自己
  you → ●     ●      -       -      看 Do + you
 have → ●     ●      ●       -      看 Do + you + have
 time → ●     ●      ●       ●      看所有 token  ← 包含全部信息
```

**最后一个 token 的上下文向量包含了整个输入的信息**，所以用它来做分类决策。

```python
# 模型输出 logits 的形状: [batch_size, seq_len, 2]
# 只取最后一个位置的输出
logits = model(input_ids)[:, -1, :]  # shape: [batch_size, 2]
```

### 6.2 计算损失

使用**交叉熵损失**（Cross-Entropy Loss）：

```python
loss = torch.nn.functional.cross_entropy(logits, targets)
```

### 6.3 计算准确率

```
logits:     [[2.1, -0.5], [-1.0, 3.2], [0.3, 0.1]]
              ↓ argmax      ↓ argmax     ↓ argmax
预测:       [0, 1, 0]       ← 每行取最大值的索引
真实标签:   [0, 1, 1]
              ↓
比较:       [✓, ✓, ✗]      → 准确率 = 2/3 ≈ 66.7%
```

```python
def calc_accuracy_loader(data_loader, model, device, num_batches=None):
    model.eval()
    correct, total = 0, 0

    for i, (input_ids, targets) in enumerate(data_loader):
        if num_batches and i >= num_batches:
            break
        input_ids, targets = input_ids.to(device), targets.to(device)
        with torch.no_grad():
            logits = model(input_ids)[:, -1, :]  # 最后一个 token
        predictions = torch.argmax(logits, dim=-1)
        correct += (predictions == targets).sum().item()
        total += targets.shape[0]

    return correct / total
```

> **Softmax 是单调函数**，不会改变大小顺序，所以 `argmax(logits)` 和 `argmax(softmax(logits))` 结果相同——不需要先 softmax 再 argmax。

## 九、在新数据上使用

### 9.1 推理函数

对新短信进行分类：

```python
def classify_text(text, model, tokenizer, device, max_length, pad_token_id=50256):
    model.eval()

    # 1. 分词
    input_ids = tokenizer.encode(text)

    # 2. 截断（如果超过 max_length）
    supported_context = model.pos_emb.weight.shape[0]  # 上下文长度
    max_length = min(max_length, supported_context)
    input_ids = input_ids[:max_length]

    # 3. 填充（如果短于 max_length）
    pad_count = max_length - len(input_ids)
    input_ids = input_ids + [pad_token_id] * pad_count

    # 4. 转为 tensor，添加 batch 维度
    input_tensor = torch.tensor(
        input_ids, device=device
    ).unsqueeze(0)  # shape: [1, max_length]

    # 5. 前向传播，取最后一个 token 的 logits
    with torch.no_grad():
        logits = model(input_tensor)[:, -1, :]

    # 6. argmax 得到预测类别
    predicted_label = torch.argmax(logits, dim=-1).item()
    return "spam" if predicted_label == 1 else "not spam"
```

### 9.2 测试示例


## 常见问题

**Q：为什么要用最后一个 token 而不是第一个？**
A：因果注意力掩码使得最后一个 token 的上下文向量包含了**所有之前 token 的信息**，而第一个 token 只包含自身信息。

**Q：为什么验证集和测试集要用训练集的 max_length？**
A：这是最佳实践。使用测试数据来确定参数相当于"数据泄露"——在真实场景中，你在推理时不会提前知道新数据的长度。

**Q：二分类为什么不用 1 个输出节点 + sigmoid？**
A：用 2 个输出节点 + softmax 更通用，代码可以直接扩展到多分类（3 类、4 类、N 类），只需要改 `num_classes`。

**Q：为什么冻结大部分层只训练最后几层？**
A：预训练模型已经学会了提取文本特征。对于简单任务，大部分知识可以复用，只需微调少量层来适配新任务。训练越少层越快，且不容易过拟合。

**Q：训练和推理时 padding token 会影响结果吗？**
A：不会。因为 padding token 都在序列末尾，而我们只使用**最后一个非 padding token**（即最后一个有意义的 token）的输出。模型在训练过程中会学会忽略 padding。
