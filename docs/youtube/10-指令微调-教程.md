### 1.2 Alpaca JSON 格式

每条数据是一个 Python 字典，包含三个字段：

| 字段 | 说明 | 示例 |
|------|------|------|
| `instruction` | 任务指令 | "Identify the correct spelling of the following word" |
| `input` | 输入内容（可为空） | "occasional" |
| `output` | 期望输出 | "The correct spelling is: occasional" |

```json
{
  "instruction": "Identify the correct spelling of the following word",
  "input": "occasional",
  "output": "The correct spelling is: occasional."
}
```

> **注意**：`input` 字段可以为空。有些指令不需要额外输入（如 "Write a short poem about the sea"）。

### 1.3 Alpaca Prompt 模板

将 JSON 条目转换为 LLM 能理解的纯文本格式：

```python
def format_input(entry):
    instruction_text = (
        f"Below is an instruction that describes a task. "
        f"Write a response that appropriately completes the request."
        f"\n\n### Instruction:\n{entry['instruction']}"
    )
    input_text = (
        f"\n\n### Input:\n{entry['input']}" if entry["input"] else ""
    )
    return instruction_text + input_text
```

转换后的文本格式：

```
Below is an instruction that describes a task. Write a response
that appropriately completes the request.

### Instruction:
Identify the correct spelling of the following word

### Input:
occasional

### Response:
The correct spelling is: occasional.
```

> **其他格式参考**：Microsoft Phi-3 使用更简洁的格式（user/assistant 标签，无样板文本）。更短的 prompt 意味着更少的 token，推理更快、更便宜。

```
├── 测试集: 10% = 110 条
```

## 二、数据集与批处理：五个关键步骤

这是**本章最关键的部分**。虽然看起来步骤不多，但直接影响模型能否正确学习。

### 2.1 五步流水线总览

```
┌─────────────────────────────────────────────────────────┐
│            数据批处理五步流水线                              │
│                                                          │
│  Step 2.1: 格式化 Prompt 模板                              │
│  ┌───────────┐                                           │
│  │ JSON 条目  │ → format_input() → 纯文本字符串            │
│  └───────────┘                                           │
│        ↓                                                  │
│  Step 2.2: Tokenize                                      │
│  ┌───────────┐                                           │
│  │ 纯文本     │ → tiktoken → Token ID 列表                │
│  └───────────┘                                           │
│        ↓                                                  │
│  Step 2.3: Padding（按批次）                               │
│  ┌───────────┐                                           │
│  │ Token IDs  │ → pad to longest in batch                │
│  └───────────┘                                           │
│        ↓                                                  │
│  Step 2.4: 创建 Target IDs（偏移一位）                     │
│  ┌───────────┐                                           │
│  │ Input IDs  │ → shift by 1 → Target IDs                │
│  │ [0,1,2,3]  │           [1,2,3,pad]                    │
│  └───────────┘                                           │
│        ↓                                                  │
│  Step 2.5: Mask 多余 Padding Tokens                       │
│  ┌───────────┐                                           │
│  │ Target IDs │ → padding → -100 (除第一个)               │
│  └───────────┘                                           │
└─────────────────────────────────────────────────────────┘
```

### 2.2 InstructionDataset 类

```python
import tiktoken

tokenizer = tiktoken.get_encoding("gpt2")

class InstructionDataset(torch.utils.data.Dataset):
    def __init__(self, data, tokenizer):
        self.data = data
        self.encoded_texts = []

        for entry in data:
            # Step 2.1: 格式化指令 + 拼接响应
            instruction_plus_input = format_input(entry)
            response_text = f"\n\n### Response:\n{entry['output']}"
            full_text = instruction_plus_input + response_text

            # Step 2.2: Tokenize
            self.encoded_texts.append(
                tokenizer.encode(full_text)
            )

    def __getitem__(self, index):
        return self.encoded_texts[index]

    def __len__(self):
        return len(self.data)
```

> **关键点**：训练时包含 response（模型要学习的内容）；推理时只提供 instruction + input，不包含 response。

### 2.3 Padding 策略：按批次填充

与第 6 章不同，这里**按每个批次中最长的序列进行填充**，而非整个数据集：

```
批次 1: 最长 80 tokens  → 所有样本 pad 到 80
批次 2: 最长 1000 tokens → 所有样本 pad 到 1000
批次 3: 最长 500 tokens → 所有样本 pad 到 500
```

**优点**：避免过多的 padding tokens，提高效率。

### 2.4 Target IDs：偏移一位

与第 5 章预训练相同的概念——模型预测下一个 token：

```
Input IDs:   [0, 1, 2, 3, pad, pad, pad]
Target IDs:  [1, 2, 3, pad, pad, pad, pad]
                  ↑ 移除第一个，末尾加一个 pad
```

### 2.5 Mask Padding：-100 忽略值

将**多余的 padding tokens** 替换为 `-100`（CrossEntropyLoss 的默认忽略值）：

```
Target IDs (mask前):  [1, 2, 3, 50256, 50256, 50256, 50256]
Target IDs (mask后):  [1, 2, 3, 50256, -100,   -100,   -100]
                              ↑ 保留一个     ↑ 其余替换
```

**为什么保留一个 padding token？** 让模型学会在响应结束时生成 end-of-text token (50256)，这样推理时模型能主动停止生成。

### 2.6 -100 的验证

```python
# 简单验证 -100 在 CrossEntropyLoss 中被忽略
logits_1 = torch.tensor([[-1.0, 1.0], [-0.5, 1.5]])
targets_1 = torch.tensor([0, 1])
loss_1 = torch.nn.functional.cross_entropy(logits_1, targets_1)

# 添加一个 target=-100 的条目
logits_2 = torch.tensor([[-1.0, 1.0], [-0.5, 1.5], [-0.6, 1.6]])
targets_2 = torch.tensor([0, 1, -100])  # -100 被忽略
loss_2 = torch.nn.functional.cross_entropy(logits_2, targets_2)

# loss_1 == loss_2  （第三个条目被忽略）
```

### 2.7 Custom Collate Function

```python
from functools import partial

def custom_collate_fn(
    batch,
    pad_token_id=50256,
    ignore_index=-100,
    allowed_max_length=None,
    device="cpu"
):
    # 找到批次中最长的序列
    batch_max_length = max(len(item) for item in batch)

    inputs_lst, targets_lst = [], []

    for item in batch:
        # --- Step 2.3: Padding ---
        new_item = item.copy()
        # 添加 padding（+1 是为了 target 偏移）
        padded = (
            new_item + [pad_token_id] *
            (batch_max_length - len(new_item))
        )

        # --- Step 2.5: 截断超长序列 ---
        if allowed_max_length is not None:
            padded = padded[:allowed_max_length]

        # --- Step 2.4: 创建 Target（偏移一位）---
        inputs = torch.tensor(padded[:-1])
        targets = torch.tensor(padded[1:])

        # --- Step 2.5: Mask 多余 padding ---
        mask = targets == pad_token_id
        indices = torch.nonzero(mask).squeeze()
        if indices.numel() > 1:
            # 保留第一个 pad token，其余替换为 -100
            targets[indices[1:]] = ignore_index

        inputs_lst.append(inputs)
        targets_lst.append(targets)

    inputs_tensor = torch.stack(inputs_lst).to(device)
    targets_tensor = torch.stack(targets_lst).to(device)
    return inputs_tensor, targets_tensor
```

### 2.8 使用 functools.partial 定制化

```python
from functools import partial

device = torch.device("cuda" if torch.cuda.is_available() else "cpu")

customized_collate_fn = partial(
    custom_collate_fn,
    device=device,
    allowed_max_length=1024
)
```

> `functools.partial` 预填充函数参数，简化后续调用。

## 四、模型评估

### 4.1 评估的挑战

指令微调的评估比分类任务困难得多：

| 分类微调（Ch6） | 指令微调（Ch7） |
|----------------|----------------|
| 固定标签（spam/not spam） | 自由格式文本 |
| 精确计算准确率 | 无唯一正确答案 |
| "50/100 正确 = 50% 准确率" | 同义词、不同表述都算对 |

```
问题: "What is the synonym for happy?"
可能正确答案: "joyful", "glad", "cheerful", "pleased", ...
```

## 五、DPO 偏好对齐（进阶）

### 5.1 为什么需要偏好对齐？

指令微调教模型**如何回答**，偏好对齐教模型**回答什么更好**：

```
指令微调 (Ch7)          偏好对齐 (Bonus)
"回答正确"              "回答得更好"

例如两个正确答案：        偏好对齐会选择：
A: "正确拼写是 xxx"     → A（更友好）
B: "你连这个都不会？"   → B 被拒绝
```

### 5.2 偏好数据格式

```
┌──────────────────────────────────────────┐
│            偏好数据条目                     │
│                                           │
│  instruction: "识别以下单词的正确拼写"        │
│  input: "occasional"                      │
│  chosen (偏好):  "正确拼写是 occasional。"   │
│  rejected (拒绝): "这是基本的，应该是xxx。"   │
└──────────────────────────────────────────┘
```

### 5.3 DPO vs RLHF

| 特性 | DPO | RLHF (PPO) |
|------|-----|------------|
| 是否需要独立的 Reward Model | 不需要 | 需要 |
| 计算内存 | 较少（1个LLM） | 较多（2个LLM） |
| 实现复杂度 | 较简单 | 较复杂 |
| 性能 | 略低但可迭代更多数据 | 理论上更好 |
| 流行度 | 越来越流行 | 传统方法 |

```
DPO:  只需要 policy model（1 个 LLM）
RLHF: 需要 policy model + reward model（2 个 LLM）
```

## 关键概念速查表

| 概念 | 说明 |
|------|------|
| **Alpaca 格式** | instruction + input + output 三字段 JSON 格式 |
| **Prompt 模板** | 将 JSON 转换为 LLM 可读的纯文本格式 |
| **Response-only Loss** | 训练时只在响应部分计算损失，指令部分不参与 |
| **-100 忽略值** | CrossEntropyLoss 默认忽略 target=−100 的位置 |
| **批次级 Padding** | 每个 batch 独立 padding 到该 batch 最长序列 |
| **End-of-text Token** | 50256，教模型何时停止生成 |
| **functools.partial** | 预填充函数参数，创建定制化的 collate 函数 |
| **DPO** | Direct Preference Optimization，不需要独立 reward model 的对齐方法 |
| **RLHF** | Reinforcement Learning from Human Feedback，传统对齐方法 |
| **LLM-as-a-Judge** | 用 LLM 评估另一个 LLM 的输出质量 |

## 常见问题

**Q: 第 7 章和第 6 章有依赖关系吗？**

A: 没有。两者都从同一个预训练模型出发（分叉关系），可以独立学习。第 6 章做分类微调，第 7 章做指令微调。

**Q: 为什么用 -100 而不是其他数字？**

A: -100 是 PyTorch `CrossEntropyLoss` 的默认 `ignore_index`。你可以用其他值（如 99 或 -1），但需要同步修改损失函数中的 `ignore_index` 参数。

**Q: 为什么保留一个 padding token 不 mask？**

A: 让模型学会在响应结束时生成 end-of-text token (50256)。推理时，当模型生成 50256 就停止，无需等到最大 token 数。

**Q: 为什么不用第 6 章的 loss 函数？**

A: 第 6 章的 `calc_loss_batch` 只基于**最后一个 token** 计算损失（适用于分类任务）。指令微调需要**所有 tokens** 都参与损失计算（与第 5 章预训练相同）。

**Q: 实际生产中需要多少指令微调数据？**

A: 通常至少 100,000 条。本教程仅用 1,100 条是为了教育目的（5-10 分钟即可训练完成）。

**Q: DPO 是必须的吗？**

A: 不是。指令微调后的模型已能正常响应。DPO 是可选的进一步优化，用于对齐偏好（更友好、更安全）。
